Objectives

The objective of this chapter is to discuss processes and techniques for developing highly dependable systems. When you have read this chapter you will:

  • Understand how system dependability can be achieved by using redundant and diverse components
  • Know how dependable software processes contribute to the development of dependable software
  • Understand how different architectural styles may be used to implement software redundancy and diversity
  • Be aware of good programming practice that should be used in dependable systems engineering

Introduction

The use of software engineering techniques, better programming languages, and better quality management has led to significant improvements in dependability for most software. Nevertheless, system failures may still occur that affect the system's availability or lead to incorrect results being produced.

Key Terms

Critical Systems

Systems where failure can lead to significant losses

Fault Avoidance

Minimizing faults during development

Fault Tolerance

System's ability to continue operating despite faults

Approaches to Dependable Software

Fault Avoidance

Using development approaches that help avoid errors

Fault Detection

Discovering and removing faults before deployment

Fault Tolerance

Managing faults at runtime to prevent failure

1. Redundancy and Diversity

Redundancy and diversity are fundamental strategies for enhancing the dependability of any type of system.

Redundancy

Spare capacity included in a system that can be used if part of that system fails.

Example:

Keeping spare light bulbs at home for when one fails.

Diversity

Redundant components of the system are of different types, increasing the chances they won't fail in the same way.

Example:

Using different types of locks on a door for security.

Real-World Applications

Server Redundancy

Systems with high availability requirements use redundant servers that automatically take over if a primary server fails.

Aircraft Systems

The Airbus 340 flight control system uses both hardware and software diversity with redundant components.

Important Consideration

While redundancy and diversity improve dependability, they also increase system complexity. This can make systems harder to understand and verify, potentially introducing new sources of errors.

2. Dependable Process

Dependable software processes are designed to produce dependable software. A company using a dependable process can be sure that the process has been properly enacted and documented.

Characteristics of Dependable Processes

Documentable
The process should have a defined process model that sets out the activities in the process and the documentation that is to be produced during these activities.
Standardized
A comprehensive set of software development standards covering software production and documentation should be available.
Auditable
The process should be understandable by people apart from process participants, who can check that process standards are being followed and make suggestions for process improvement.
Diverse
The process should include redundant and diverse verification and validation activities.
Robust
The process should be able to recover from failures of individual process activities.

Attributes of Dependable Processes

Explicitly Defined

Has a defined process model with collected data demonstrating all necessary steps were followed.

Repeatable

Does not rely on individual interpretation; can be repeated across projects with different teams.

Key Activities in Dependable Processes

Requirements reviews to check completeness and consistency

Formal specification using mathematical models

Design and program inspections by different people

Static analysis using automated checks on source code

Comprehensive test planning and management

Process Comparison

Traditional Plan-Based
  • Extensive documentation
  • Formal change management
  • Less flexible to changes
Agile Approaches
  • Adaptable to changes
  • Faster feedback cycles
  • Less formal documentation

3. Dependable System Architectures

Dependable systems require architectures designed for fault tolerance, often including redundant components and mechanisms to switch control when failures occur.

Architecture Styles for Dependability

Protection Systems

Specialized systems that monitor another system and take action if problems are detected.

Example: A train protection system that automatically applies brakes if the train passes a red signal.

Self-monitoring Architectures

Systems designed to monitor their own operation and take action if problems are detected.

Key Feature: Computations are carried out on separate channels and compared.

N-version Programming

Multiple versions of software are developed independently from the same specification.

Application: Used in railway signaling, aircraft systems, and reactor protection systems.

Achieving Software Diversity

Diversity Policies
  • 1. Using different design methods
  • 2. Implementing in different programming languages
  • 3. Using different tools and development environments
  • 4. Implementing different algorithms for the same functionality
Challenges
  • 1. Common specification errors
  • 2. Similar cultural and educational backgrounds
  • 3. High development costs
  • 4. Increased system complexity

4. Dependable Programming

Good programming practices help reduce faults in delivered systems. These guidelines can be applied regardless of the programming language used.

Programming Guidelines for Dependability

Control Information Visibility

Program components should only access data they need. Implement data structures as abstract data types to hide internal representation.

Check All Inputs for Validity

Always validate inputs as soon as they are read. Implement range checks, size checks, representation checks, and reasonableness checks.

Provide Handlers for All Exceptions

Define exception handlers for all possible exceptions. Exception handlers can signal higher-level components, carry out alternative processing, or pass control to runtime systems.

Limit Use of Problematic Constructs

Be cautious with pointers, dynamic memory allocation, recursion, interrupts, and other constructs that can introduce vulnerabilities.

Provide Restart Capabilities

For long transactions or computationally intensive systems, implement checkpointing to save progress and allow restart from the last checkpoint.

Check Array Bounds

Always ensure array indices are within bounds to prevent buffer overflow vulnerabilities.

Include Timeouts for External Calls

When calling external components, implement timeouts to prevent indefinite waiting if the component fails to respond.

Name Constants Representing Real-world Values

Use named constants instead of literal values to reduce errors and make maintenance easier.

KEY POINTS

Dependability in a program can be achieved by avoiding the introduction of faults, by detecting and removing faults before system deployment, and by including fault-tolerance facilities that allow the system to remain operational after a fault has caused a system failure.

The use of redundancy and diversity in hardware, software processes, and software systems is essential to the development of dependable systems.

The use of a well-defined, repeatable process is essential if faults in a system are to be minimized. The process should include verification and validation activities at all stages, from requirements definition through to system implementation.

Dependable system architectures are system architectures that are designed for fault tolerance. There are a number of architectural styles that support fault tolerance including protection systems, self-monitoring architectures, and N-version programming.

Software diversity is difficult to achieve because it is practically impossible to ensure that each version of the software is truly independent.

Dependable programming relies on the inclusion of redundancy in a program to check the validity of inputs and the values of program variables.

Some programming constructs and techniques, such as go-to statements, pointers, recursion, inheritance, and floating-point numbers, are inherently error prone. You should try to avoid these constructs when developing dependable systems.
