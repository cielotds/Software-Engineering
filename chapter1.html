Part 1

Integration with the Web

The web contains a very large and continually expanding body of software-engineering material whose quality and depth vary; to help learners we keep carefully selected, stable core material in the printed text while offering extended, up-to-date supplemental material online that explains advanced concepts, provides additional worked examples, and points to authoritative external references for deeper study. Added material on the site includes short tutorials, extended examples with code snippets, and links to community tools that illustrate the ideas in practice. These resources are organised so instructors can pick focused tasks for workshops and students can follow self-study paths that reinforce classroom learning.

http://www.SoftwareEngineering-9.com

The book's web presence supports four main functions: additional web sections linked from chapter breakout boxes; longer web chapters on specialist areas; instructor resources such as slides and case artefacts; and extended case-study information and datasets that cannot be included in print.

Audience

Readership

This book targets university and college students studying introductory and advanced software and systems engineering courses, but it is also intended as a concise reference for practising engineers, technical leads, and instructors who need a practical, up-to-date overview of engineering principles, trade-offs, and contemporary practices; readers should be familiar with basic programming and computer science terminology before using the text in depth. Instructors will find modular chapter layouts suitable for short exercises or longer project work, while practitioners can use selected sections as quick refreshers when designing or reviewing systems.

Teaching

Using Book for teaching

I have designed the book so it can be used in three types of courses: concise one‑semester introductions that cover essential concepts and practices; focused intro/intermediate courses that combine selected parts with practical exercises; and advanced courses that pair chapters with extensive web resources and case studies. Where useful, instructors can assign linked web exercises that provide datasets and worked examples; these make it straightforward to set lab work that mirrors the theory presented in class.

Objectives

By the end of this chapter you should be able to define software engineering in practical terms, recognise the main activities involved in building software systems, appreciate how different kinds of systems influence engineering decisions, and understand the professional and ethical responsibilities that accompany the practice of software engineering. These outcomes are supported by short exercises and reflective questions at the end of sections to help consolidate understanding.

  • Understand what software engineering is and why it is important.
  • Understand that different system types require different engineering techniques.
  • Understand ethical and professional issues for software engineers.
  • Have been introduced to three case study systems used as examples.

History of Software engineering

The notion of ‘software engineering’ was first proposed in 1968 during discussions of the ‘software crisis’. Over subsequent decades, techniques such as structured programming and object-oriented development emerged, along with tools and standard notations now widely used. These historical shifts show how the discipline has continually adapted to changing problem scales and development environments; reflecting on them helps explain why certain practices (for example modularity and testing) remain central today.

1.1 Professional software development

Software includes programs, documentation, and configuration needed to run systems. A professional system includes multiple programs, docs, and user documentation. In practice, the lifecycle of a professional system includes staged delivery, formal reviews, and traceable requirements to support maintenance and compliance.

1.2 Software engineering ethics

Professional societies (ACM, IEEE, BCS) publish codes of ethics encouraging public safety, competence, confidentiality, and integrity. Like other engineering disciplines, software engineering is carried out within legal and social frameworks. Engineers must accept that their work affects many people and should behave ethically. When dilemmas arise, documenting the decision process and seeking peer advice are practical steps that help balance competing responsibilities.

1.3 Case studies

Case studies illustrate different aspects and types of systems. The three used are: an embedded insulin pump control system, a patient information system for mental health care, and a wilderness weather station. Readers are encouraged to compare the design constraints and testing strategies for each case study to understand how safety, privacy and reliability trade-offs influence architecture and process choices.

Key points

  • Software engineering covers all aspects of software production.
  • Essential attributes: maintainability, dependability, efficiency, acceptability.
  • Fundamental activities: specification, development, validation, evolution.

These points form the basis for the chapter exercises and the accompanying web material which expands on practical techniques and examples.
